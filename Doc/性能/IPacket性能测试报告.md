# IPacket 数据包性能测试报告

## 一、性能概览

IPacket 体系提供四种数据包实现（OwnerPacket / MemoryPacket / ArrayPacket / ReadOnlyPacket）和 PacketHelper 扩展方法，用于网络编程和协议解析中的高性能内存管理。核心操作均在亚纳秒到十几纳秒级别。

- **ArrayPacket 综合最优**：构造 / GetSpan / TryGetArray 被 JIT 完全内联（< 0.02 ns），struct Slice 零分配（2.6 ns / 0B），Indexer 读写仅 0.2 ns，建议作为通用首选
- **OwnerPacket 适合 IO 场景**：基于 ArrayPool 池化构造（6.7 ns / 0B），但 32 线程下 ArrayPool 竞争导致构造 98 us，是 struct 实现的 12 倍
- **struct 天然多线程友好**：三种 struct 实现 32 线程构造仅 ~8 us / 3.6 KB，增长倍数一致（3.1×），无锁竞争
- **Slice 装箱是内存主开销**：通过 IPacket 接口返回 Slice 导致 struct 装箱 32-48 B/次；代码已为 MemoryPacket / ReadOnlyPacket 补充 struct Slice 重载消除装箱
- **ToStr 是 CPU 瓶颈**：8KB 时 127 us / 127 KB，32 线程累计 481 MB；代码已提供 `WriteTo(TextWriter)` 流式替代

> **注**：以下基准数据采集于代码优化前。MemoryPacket Indexer 缓存、struct Slice 重载、WriteTo API 等优化已在代码中实现，待后续重新采集验证。

## 四、结果分析

**类型间性能差异**（Size=1024 单线程）：ArrayPacket 构造比 OwnerPacket 快约 670 倍（JIT 内联 vs 6.69 ns），GetSpan 快约 1125 倍。Slice 上 ArrayPacket struct 版本 2.6 ns / 0B，MemoryPacket 和 ReadOnlyPacket 通过 IPacket 接口分别为 6.16 ns / 48B 和 4.69 ns / 32B（均因装箱）。Indexer 上 ArrayPacket（0.24 ns）直接数组访问，MemoryPacket（1.66 ns）需经 `Memory.Span` 间接访问，慢约 7 倍。

**多线程扩展性**：三种 struct 实现 1→32 线程构造增长完全一致（3.1×），仅受线程调度开销影响，无锁竞争。OwnerPacket 构造增长 7.4×（ArrayPool 竞争），Slice 增长最陡（15.1×），每次 48B 装箱叠加 GC 压力。32 线程 Slice 分配：ArrayPacket 4.5 KB vs OwnerPacket 3006 KB（668 倍差距）。

**PacketHelper 特征**：Append / TryGetSpan / GetStream / ToSegment（单包）均为常量时间，不受 Size 影响。ToArray / Clone / ToStr 随 Size 线性增长。ToStr 是最慢操作（8KB 单线程 127 us / 127 KB，32 线程 51 ms / 481 MB），链式包约为单包 2 倍。链式包 ToSegment 开销随 Size 急剧增长（64B 时 5.6×，8KB 时 117×），因需合并到连续数组。ToHex 仅编码前 32 字节，链式包开销稳定（1.2-1.3×）。

## 五、瓶颈与优化建议

| 优先级 | 瓶颈 | 现象（实测数据） | 优化方向 | 预期收益 |
|--------|------|----------------|---------|---------|
| P0 | Slice IPacket 装箱 | MemoryPacket 48B、ReadOnlyPacket 32B、ArrayPacket(IPacket) 40B/次 | 已添加 struct Slice 重载（MemoryPacket / ReadOnlyPacket） | 降至 0B/次，待重测验证 |
| P0 | ToStr 大包内存分配 | 8KB→127 us / 127 KB，32T→481 MB | 已添加 `WriteTo(TextWriter)` 流式输出 | 大包场景消除中间 String，待重测验证 |
| P0 | MemoryPacket Indexer 间接访问 | 1.66 ns（ArrayPacket 0.24 ns，慢 7×） | 已缓存底层数组引用（`MemoryMarshal.TryGetArray`） | 预期降至 ~0.24 ns，待重测验证 |
| P1 | OwnerPacket 多线程 ArrayPool 竞争 | 32T 构造 98 us，struct 实现 ~8 us（慢 12×） | ThreadLocal 缓存或 Per-Thread 池替代 `ArrayPool.Shared` | 预期降至 ~8 us |

P0 瓶颈均已在代码层面完成优化，待重新采集基准数据验证实际收益。剩余 P1 中 OwnerPacket ArrayPool 竞争是最显著的未解决瓶颈，主要影响高并发网络 IO 场景。此外，链式包 ToSegment 合并拷贝问题已通过新增 `ToSegments` 方法（返回分段列表，避免合并到连续数组）提供零拷贝替代方案。

| 实现类型 | 推荐场景 |
|---------|---------|
| ArrayPacket | 通用首选，零拷贝切片，最快索引访问 |
| OwnerPacket | 网络 IO 频繁申请 / 释放缓冲区，需池化内存管理 |
| MemoryPacket | 包装 `Memory<Byte>` 的临时场景 |
| ReadOnlyPacket | 多线程共享只读数据，线程安全 |
